## The Codex: Python Import Mysteries — Explained

This module is essentially a **deep dive into how Python's import system works**, using an "alchemist lab" theme to make it engaging. Let me break down what they're actually asking:

---

## The 4 Core Concepts (The "4 Sacred Mysteries")

### Mystery 1 — Package Initialization (`__init__.py`)
- A folder becomes a Python **package** only when it has an `__init__.py` file
- That file controls **what gets exposed** when someone does `import alchemy`
- You can hide functions, create aliases, or re-export things selectively

### Mystery 2 — Import Pathways
- `import module` vs `from module import something` — two different styles
- Accessing files **directly** (like `alchemy.elements`) vs through a **package interface** (`import alchemy`)
- Understanding that Python searches `sys.path` to find modules

### Mystery 3 — Absolute vs Relative Imports
- **Absolute**: `from alchemy.potions import healing_potion` — full path from project root
- **Relative**: `from .potions import healing_potion` — relative to current file's location
- Both reach the same thing, but have different use cases (portability vs clarity)

### Mystery 4 — Circular Dependencies
- When module A imports from B, and B imports from A → **infinite loop → crash**
- They want you to intentionally **create** this explosion (dark magic 💥) and also **solve** it (light magic ✅)

## DataDeck: Abstract Card Architecture — Explained

This module is about **3 advanced OOP design patterns** in Python, using a creature card game as the playground. Here's what's actually going on:

---

## The 3 Core Patterns

### Pattern 1 — Abstract Factory (Exercise 0)
The problem: you want to create *families* of related objects without hardcoding which specific classes to use.

```
CreatureFactory (abstract)
├── FlameFactory → creates Flameling + Pyrodon
└── AquaFactory  → creates Aquabub + Torragon
```

The caller only talks to the factory, never to the concrete classes directly. That's why `__init__.py` must **hide** the concrete Creatures and **only expose factories**.

### Pattern 2 — Mixin / Multiple Inheritance for Capabilities (Exercise 1)
The problem: you want to add optional behaviors (heal, transform) to some Creatures but not all, without polluting the base class.

```
Creature (abstract)
HealCapability (abstract)   ← totally separate hierarchy
TransformCapability (abstract)

Sproutling inherits from BOTH Creature + HealCapability
Shiftling  inherits from BOTH Creature + TransformCapability
```

The key insight: capabilities are **independent** of Creature so they could later be applied to non-Creature things too.

### Pattern 3 — Strategy Pattern (Exercise 2)
The problem: different Creatures fight differently depending on their capabilities. Instead of hardcoding `if isinstance(creature, HealCapability)` everywhere, you encapsulate the *behavior* into a separate strategy object.

```
BattleStrategy (abstract)
├── NormalStrategy    → just attack (works for any Creature)
├── DefensiveStrategy → attack + heal (only valid for HealCapability)
└── AggressiveStrategy→ transform + attack + revert (only valid for TransformCapability)
```

The tournament doesn't care *what* the Creature is — it just calls `strategy.act(creature)`.

```

  Creature (abstract) + concrete: Flameling, Pyrodon, Aquabub, Torragon
```

## C'est quoi le problème qu'on essaye de résoudre?

Imagine t'as un jeu de cartes avec des créatures. Tu veux créer des centaines de créatures différentes. Si t'écris une classe pour chaque créature sans organisation → **code spaghetti, impossible à maintenir.**

La solution = **Design Patterns** (des recettes d'organisation de code).

---

## Pattern 1 — Abstract Class (la base de tout)

Une **classe abstraite** c'est comme un **contrat** :
- Elle dit *"toute classe qui hérite de moi DOIT implémenter ces méthodes"*
- Tu peux pas l'instancier directement (pas de `Creature()`)

```python
from abc import ABC, abstractmethod

class Creature(ABC):          # ABC = Abstract Base Class
    
    def describe(self):       # méthode CONCRÈTE → même pour tout le monde
        return f"{self.name} is a {self.type} type Creature"
    
    @abstractmethod
    def attack(self):         # méthode ABSTRAITE → chaque créature DOIT la définir
        pass
```

Si tu fais `Creature()` → **erreur**. C'est fait exprès, c'est juste un modèle.

---

## Pattern 2 — Abstract Factory (Exercise 0)

**Le problème :** Les créatures viennent en **familles** (feu, eau...). Chaque famille a une version de base et une version évoluée.

**Sans factory :**
```python
# Le code qui utilise les créatures doit savoir TOUS les noms
c1 = Flameling()   # version de base feu
c2 = Pyrodon()     # version évoluée feu
c3 = Aquabub()     # version de base eau
```

**Avec factory :**
```python
# Le code qui utilise les créatures parle SEULEMENT à la factory
factory = FlameFactory()
c1 = factory.create_base()     # il sait pas que c'est Flameling
c2 = factory.create_evolved()  # il sait pas que c'est Pyrodon
```

La factory = **intermédiaire** entre ton code et les classes concrètes. Tu changes de famille ? Tu changes juste la factory.

```
CreatureFactory (abstraite)
├── FlameFactory → create_base() = Flameling, create_evolved() = Pyrodon
└── AquaFactory  → create_base() = Aquabub,   create_evolved() = Torragon
```

## Pattern 3 — Mixins / Multiple Inheritance (Exercise 1)

**Le problème :** Certaines créatures peuvent se soigner, d'autres peuvent se transformer. Mais pas toutes. Comment ajouter ces capacités sans tout casser ?

**Mauvaise idée :**
```python
class Creature(ABC):
    def heal(self): ...      # ❌ pas toutes les créatures peuvent se soigner!
    def transform(self): ... # ❌ pareil
```

**Bonne idée — Mixin (capacité séparée) :**
```python
class HealCapability(ABC):      # classe INDÉPENDANTE
    @abstractmethod
    def heal(self): pass

class TransformCapability(ABC): # classe INDÉPENDANTE
    @abstractmethod
    def transform(self): pass
    @abstractmethod
    def revert(self): pass

# Une créature qui peut se soigner hérite des DEUX
class Sproutling(Creature, HealCapability):
    def attack(self): ...   # obligé car Creature l'exige
    def heal(self): ...     # obligé car HealCapability l'exige

# Une créature normale hérite juste de Creature
class Flameling(Creature):
    def attack(self): ...   # juste ça
```

Le mot **Mixin** = une classe qu'on "mélange" dans une autre pour lui ajouter des capacités.

## Pattern 4 — Strategy Pattern (Exercise 2)

**Le problème :** Dans un tournoi, chaque créature se bat différemment selon ses capacités. Comment organiser ça proprement ?

**Mauvaise idée :**
```python
def battle(creature):
    if isinstance(creature, HealCapability):
        creature.attack()
        creature.heal()
    elif isinstance(creature, TransformCapability):
        creature.transform()
        creature.attack()
        creature.revert()
    else:
        creature.attack()
```
→ Chaque fois qu'on ajoute une capacité, on modifie cette fonction. **Mauvais.**

**Bonne idée — Strategy :**
```python
class BattleStrategy(ABC):
    @abstractmethod
    def act(self, creature): pass      # comment se battre
    
    @abstractmethod
    def is_valid(self, creature): pass # est-ce que cette stratégie marche pour cette créature?

class NormalStrategy(BattleStrategy):
    def is_valid(self, creature): return True  # marche pour tout le monde
    def act(self, creature): creature.attack()

class DefensiveStrategy(BattleStrategy):
    def is_valid(self, creature): return isinstance(creature, HealCapability)
    def act(self, creature): 
        creature.attack()
        creature.heal()
```

Maintenant le tournoi fait juste :
```python
strategy.act(creature)  # il sait pas ce que c'est, il s'en fout
```

```python
from abc import ABC, abstractmethod
from ex0.creatures import Creature
from ex0.fire import Flameling, Pyrodon
from ex0.water import Aquabub, Torragon


# ABSTRACT FACTORY — l-contrat
class CreatureFactory(ABC):

    @abstractmethod
    def create_base(self) -> Creature:    # kol factory Mجبورة t-implement hada
        pass

    @abstractmethod
    def create_evolved(self) -> Creature: # o hada
        pass

    def create_base(self) -> Creature:
        return Flameling()    # hna factory 3rfa chno tsni

    def create_evolved(self) -> Creature:

# CONCRETE FACTORY — dial l-ma

    def create_base(self) -> Creature:
        return Aquabub()

    def create_evolved(self) -> Creature:
        return Torragon()
```
